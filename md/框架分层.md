# 框架分层

## 1. 这份文档的用途

这份文档是后续新增功能时的代码放置依据。

以后做新功能时，先不要急着把代码写进 `main.cpp`，也不要把页面、业务逻辑、配置、资源、文案混在一个文件里。先按这份分层判断代码属于哪一层，再决定文件应该放在哪里。

当前项目已经从单文件主循环，调整成了一个缩小版的 App 框架：

- `main.cpp` 只负责系统入口
- `apps` 负责具体页面和应用
- `sys` 负责 App 生命周期、页面注册、系统配置等
- `hal` 负责硬件访问统一封装
- `ui` 负责通用 UI 参数和布局约定
- `lang` 负责显示文案
- `input`、`screen`、`u8g2_wrapper` 暂时作为底层过渡模块保留

后续新功能都应该以这个结构为前提继续扩展。

## 2. 总体分层关系

推荐按下面的依赖方向理解项目：

```text
main.cpp
  |
  v
sys/AppManager
  |
  v
apps/App 页面
  |
  +--> ui        通用布局、颜色、字体、主题参数
  +--> lang      文案
  +--> sys       配置、资源、注册、系统服务
  +--> hal       显示、输入、硬件能力
          |
          +--> input
          +--> screen
          +--> u8g2_wrapper
```

依赖方向应该尽量保持单向：

- `main.cpp` 可以调用 `HAL_Init()` 和 `appManager.begin()/run()`
- `sys` 可以调度 `apps`
- `apps` 可以使用 `hal`、`ui`、`lang`、`sys`
- `hal` 可以封装 `input`、`screen`、底层库
- `ui` 不应该依赖具体业务页面
- `lang` 不应该依赖具体页面状态

不推荐反向依赖：

- `hal` 不应该知道 `AppGacha`、`AppMainMenu`
- `ui` 不应该知道抽卡、设置、统计这类业务
- `lang` 不应该直接调用页面代码
- `main.cpp` 不应该直接处理某个功能的状态机

## 4. `main.cpp` 层

### 职责

`main.cpp` 只做系统入口，不承载具体功能。

它应该只保留：

- 串口初始化
- `HAL_Init()`
- `appManager.begin()`
- `appManager.run()`
- 必要的全局启动日志

### 不应该放

- 不放具体页面绘制代码
- 不放抽卡、设置、统计等业务状态机
- 不放按钮点击后的具体功能判断
- 不放菜单滚动动画
- 不放卡池数据
- 不放配置读写逻辑

### 判断标准

如果一段代码只有某个功能会用，它就不应该放进 `main.cpp`。

## 5. `apps` 层

### 职责

`apps` 是具体页面和应用所在的位置。

每一个可以被打开、返回、暂停、恢复的页面，都应该是一个 App。

当前已有：

- `app_standby.cpp`
- `app_main_menu.cpp`
- `app_gacha.cpp`
- `app_menu_base.h`

### 放什么

适合放在 `apps` 的内容：

- 页面状态机
- 页面输入响应
- 页面绘制流程
- 页面之间的跳转意图
- 某个功能的入口页面
- 某个功能的结果页面
- 某个功能的设置页面

例如抽卡功能：

- 抽卡首页放在 `apps/sources/app_gacha.cpp`
- 抽卡统计页可以放在 `apps/sources/app_gacha_stats.cpp`
- 抽卡设置页可以放在 `apps/sources/app_gacha_settings.cpp`

### 不应该放

- 不直接操作底层屏幕对象
- 不直接操作 EC11 原始输入对象
- 不直接写 LittleFS 细节
- 不直接塞大量静态资源数据
- 不把通用菜单动画复制到每个页面

### 新增 App 的基本步骤

1. 在 `src/apps/sources/` 新建 `app_xxx.cpp`
2. 继承 `AppBase` 或 `AppMenuBase`
3. 在 `src/sys/includes/sys/app_registry.h` 增加 `AppId`
4. 在 `src/sys/sources/app_registry.cpp` 注册页面对象
5. 在入口页面里通过 `appManager.push()` 或 `appManager.launch()` 打开

### `AppBase` 和 `AppMenuBase` 怎么选

使用 `AppBase` 的情况：

- 页面完全自定义
- 页面不是菜单结构
- 需要独立动画、独立状态机
- 例如抽卡扫描动画页、结果页、待机页

使用 `AppMenuBase` 的情况：

- 页面本质是菜单
- 有多个菜单项
- 需要复用当前滚轮菜单 UI
- 例如主菜单、抽卡子菜单、设置菜单、统计菜单

## 6. `sys` 层

### 职责

`sys` 是系统框架层，不属于某个具体页面。

当前已有：

- `app_base.h`
- `app_manager.h / app_manager.cpp`
- `app_registry.h / app_registry.cpp`
- `sys_constants.h`

后续可以继续增加：

- `sys_fs.h / sys_fs.cpp`
- `sys_config.h / sys_config.cpp`
- `sys_res.h / sys_res.cpp`

### 放什么

适合放在 `sys` 的内容：

- App 生命周期接口
- App 跳转、返回、栈管理
- App 注册表
- 系统常量
- 文件系统初始化
- 配置读写
- 资源加载
- 全局运行时状态

### 不应该放

- 不放某个页面的绘制细节
- 不放菜单动画
- 不放具体按钮按下后某个页面如何显示
- 不放只属于抽卡页面的局部变量

### 配置和资源的建议拆法

后续接文件系统时，建议这样拆：

- `sys_fs`：只管 LittleFS 或文件系统初始化
- `sys_config`：只管配置读写，例如统计、设置项
- `sys_res`：只管资源加载，例如卡池、图片、字库索引

不要一开始把文件系统、JSON、卡池解析、统计保存全部塞进一个 App 文件。

## 7. `hal` 层

### 职责

`hal` 是硬件抽象层。

页面不应该直接关心屏幕对象是什么，也不应该直接关心 EC11 类如何读取。

当前已有：

- `hal.h`
- `hal.cpp`

### 放什么

适合放在 `hal` 的内容：

- 屏幕初始化
- 统一绘制接口
- 字体设置
- 文本宽度测量
- EC11 输入轮询
- 短按、长按、旋钮增量读取
- 未来的震动、蜂鸣器、背光控制

### 不应该放

- 不放 App 跳转
- 不放抽卡概率
- 不放菜单树
- 不放页面状态

### 判断标准

如果这段代码是在回答“硬件怎么做”，放 `hal`。

如果这段代码是在回答“功能怎么做”，不要放 `hal`。

## 8. `ui` 层

### 职责

`ui` 放通用 UI 规则，不放具体业务页面。

当前已有：

- `ui_theme.h`

### 放什么

适合放在 `ui` 的内容：

- 颜色
- 间距
- 字体选择
- 菜单布局参数
- 公共组件尺寸
- 通用绘制风格

后续可以增加：

- `ui_font_config.h`
- `ui_layout.h`
- `ui_common_draw.h`

### 不应该放

- 不放抽卡结果
- 不放菜单数据
- 不放配置读写
- 不放 App 跳转

### 判断标准

如果这段代码可以被多个页面复用，并且只和显示风格有关，放 `ui`。

## 9. `lang` 层

### 职责

`lang` 放文案。

当前已有：

- `ui_strings.h`

### 放什么

适合放在 `lang` 的内容：

- 页面标题
- 按钮提示
- 状态提示
- 菜单显示文本
- 错误提示
- 多语言字符串入口

### 不应该放

- 不放显示坐标
- 不放颜色
- 不放业务逻辑
- 不放 App 跳转

### 当前阶段建议

当前阶段先用 `ui_strings.h` 集中管理文案即可。

以后如果要做多语言，再考虑：

- `terminal_lang.h`
- `ui_strings_zh.h`
- `ui_strings_en.h`

## 10. `input`、`screen`、`u8g2_wrapper` 过渡层

### 当前状态

这些目录暂时保留为底层实现。

- `input`：EC11 原始输入实现
- `screen`：屏幕底层封装
- `u8g2_wrapper`：U8g2 与 TFT_eSPI 的适配

### 后续规则

新功能页面不要直接调用这些目录。

新功能页面应该通过 `hal` 使用它们。

也就是说：

- 页面要画文字，调用 `HAL_DrawText()`
- 页面要读旋钮，等 `AppManager` 分发 `onKnob()`
- 页面要判断短按，等 `AppManager` 分发 `onKeyShort()`

这样以后底层屏幕或输入实现变化时，不需要改所有 App。

## 11. `images` 和未来 `data`

### `images`

当前 `images` 放代码形式的图片资源，例如：

- `ak.h`

适合放：

- 小图标
- bitmap
- 暂时编译进固件的图片资源

不适合放：

- 页面逻辑
- 卡池数据
- 配置数据

### `data`

未来接 LittleFS 后，建议新增 `data/` 放运行时资源。

适合放：

- 卡池 JSON
- 配置默认值
- 多语言资源
- 可替换资源

示例：

```text
data/
├── common/
│   └── config_common.json
├── zh/
│   └── ids_zh.json
└── en/
    └── ids_en.json
```

当前阶段如果还没接 LittleFS，不要急着创建复杂 `data` 结构。

## 12. 新功能代码应该怎么放

### 情况 1：新增一个独立页面

例如新增“关于本机”页面：

- 页面文件：`src/apps/sources/app_about.cpp`
- 注册入口：`src/sys/includes/sys/app_registry.h`
- 注册实现：`src/sys/sources/app_registry.cpp`
- 菜单入口：`src/menu.cpp` 或后续独立菜单数据文件
- 文案：`src/lang/includes/lang/ui_strings.h`
- 样式：`src/ui/includes/ui/ui_theme.h`

### 情况 2：新增一个菜单型功能

例如新增“系统设置”：

- 页面继承 `AppMenuBase`
- 菜单项放页面附近，或后续拆成独立菜单数据文件
- 点击叶子节点后再进入具体页面
- 通用菜单动画继续复用 `AppMenuBase`

### 情况 3：新增一个复杂业务功能

例如抽卡：

- 页面入口：`apps/sources/app_gacha.cpp`
- 抽卡统计页面：`apps/sources/app_gacha_stats.cpp`
- 卡池资源加载：`sys/sources/sys_res.cpp`
- 统计保存：`sys/sources/sys_config.cpp`
- 文案：`lang/includes/lang/ui_strings.h`
- 颜色和布局：`ui/includes/ui/ui_theme.h`
- 硬件反馈：`hal/sources/hal.cpp`

不要把这些全部写在 `app_gacha.cpp` 里。

### 情况 4：新增硬件能力

例如蜂鸣器、震动马达、背光：

- 引脚常量：`sys/includes/sys/sys_constants.h`
- 硬件初始化和调用：`hal/includes/hal/hal.h`、`hal/sources/hal.cpp`
- 页面只调用 `HAL_*` 接口

### 情况 5：新增配置项

例如是否开启音效：

- 默认值和读写：未来放 `sys_config`
- 页面入口：设置页面 App
- 文案：`lang`
- UI：`ui`
- 不要只存在页面局部变量里

## 13. 新增 App 的模板流程

新增一个 App 时，按这个顺序做：

1. 新建 `src/apps/sources/app_xxx.cpp`
2. 定义 `class AppXxx : public AppBase`
3. 实现 `onCreate()`、`onLoop()`、`onDestroy()`、`onKnob()`、`onKeyShort()`、`onKeyLong()`
4. 文件底部暴露 `AppBase *appXxx`
5. 在 `AppId` 增加 `Xxx`
6. 在 `AppRegistry_Get()` 增加 `case AppId::Xxx`
7. 在入口处调用 `appManager.push(AppId::Xxx)` 或 `appManager.launch(AppId::Xxx)`
8. 编译验证
9. 上机验证短按、长按、返回、重进页面

## 14. `launch()` 和 `push()` 怎么选

使用 `launch()`：

- 切换到一个新的根页面
- 清空返回栈
- 例如从待机页进入主菜单
- 例如回到待机页

使用 `push()`：

- 从当前页面进入子页面
- 希望长按或返回后回到上一页
- 例如主菜单进入抽卡
- 例如设置菜单进入关于页

使用 `pop()`：

- 当前页面结束，回到上一个页面
- 例如抽卡页长按返回主菜单

## 15. 功能边界判断表

| 你要写的内容 | 应该放哪里 |
|---|---|
| 新页面 | `apps/sources/app_xxx.cpp` |
| 页面公共接口 | `sys/includes/sys/app_base.h` |
| 页面注册 | `sys/app_registry.*` |
| 页面跳转 | `sys/app_manager.*` |
| 菜单公共绘制 | `apps/includes/apps/app_menu_base.h` |
| 屏幕绘制能力 | `hal/hal.*` |
| EC11 输入能力 | `hal/hal.*`，底层继续用 `input` |
| 颜色、间距、布局参数 | `ui/ui_theme.h` |
| 文案 | `lang/ui_strings.h` |
| 系统常量、引脚 | `sys/sys_constants.h` |
| 配置读写 | 未来 `sys_config.*` |
| 文件系统初始化 | 未来 `sys_fs.*` |
| 卡池、资源加载 | 未来 `sys_res.*` |
| 图片 bitmap | `images/` |

## 16. 不建议的写法

后续新增功能时，尽量避免这些写法：

- 不要把新功能直接写进 `main.cpp`
- 不要在页面里直接创建和操作底层屏幕对象
- 不要在页面里直接读取 EC11 原始状态
- 不要把文案散落在每个页面里
- 不要在每个页面复制一份菜单滚动动画
- 不要把配置文件读写塞进具体 App
- 不要让 `hal` 反向依赖具体 App
- 不要在一个提交里同时做框架、UI、资源、业务规则四件事

## 17. 推荐的开发节奏

每做一个新功能，推荐按这个节奏：

1. 先做页面骨架
2. 注册到 `AppRegistry`
3. 从菜单能进入
4. 长按能返回
5. 再做最小业务逻辑
6. 再做显示细节
7. 再接配置或资源
8. 最后再加动画、音效、震动

这样每一步都能编译、能上机验证，也更容易知道问题是哪一步引入的。

## 18. 抽卡功能按当前分层的放置建议

抽卡后续继续完善时，建议这样拆：

```text
src/apps/sources/app_gacha.cpp          // 抽卡主页面：状态机、输入、绘制流程
src/apps/sources/app_gacha_stats.cpp    // 抽卡统计页面
src/sys/sources/sys_res.cpp             // 卡池资源加载
src/sys/sources/sys_config.cpp          // 统计数据保存
src/lang/includes/lang/ui_strings.h     // 抽卡文案
src/ui/includes/ui/ui_theme.h           // 抽卡颜色、布局参数
data/zh/ids_zh.json                     // 后续中文卡池
data/common/config_common.json          // 后续统计配置
```

当前已经有 `app_gacha.cpp` 骨架，下一步应该先在这个 App 内做最小可运行抽卡流程。

不要一开始就同时接 JSON、LittleFS、统计持久化、动画、音效。

## 19. 最小验收标准

以后每新增一个功能，至少满足下面标准再继续下一层：

- 能编译通过
- 能从菜单进入
- 短按和长按行为明确
- 能从页面返回
- 重复进入页面不会死机
- 页面不直接污染 `main.cpp`
- 新增文案集中在 `lang`
- 新增 UI 参数集中在 `ui`
- 新增硬件能力通过 `hal`

## 20. 总结

这套框架的核心不是“文件夹变多”，而是把不同性质的代码放回自己的位置。

后续新增功能时，优先按下面这句话判断：

页面放 `apps`，调度放 `sys`，硬件放 `hal`，样式放 `ui`，文案放 `lang`，资源和配置后续放 `data` 与 `sys_res/sys_config`。

只要坚持这个边界，后面继续加抽卡、统计、设置、资源加载时，项目就不会再退回到所有逻辑都堆在 `main.cpp` 的状态。
