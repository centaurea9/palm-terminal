# 功能实现

## 1. 文档目标

这份文档用于指导后续实现“干员寻访”功能。

它不是一次性把代码全部写完，而是把功能拆成和当前项目框架匹配的模块，让你以后可以照着实例项目的思路继续模仿、替换和扩展。

参考来源：

- `md/抽卡.md`
- `md/框架分层.md`
- 实例项目 `D:\project\test\Prescript_Terminal-2.79-\src\apps\sources\app_gacha.cpp`
- 实例项目 `D:\project\test\Prescript_Terminal-2.79-\src\apps\sources\app_gacha_stats.cpp`
- 实例项目 `sys_res`、`sys_config`、`app_registry`、`app_main_menu` 的抽卡相关结构

核心目标：

- 保留当前项目的 `AppManager / AppRegistry / AppBase / AppMenuBase` 分层。
- 把抽卡入口、抽卡流程、统计历史、卡池资源、配置保存分开。
- 第一版先做可运行流程，不做音效、震动、持久化、双语言。
- 后续要换卡池、改概率、加统计保存时，不需要推翻页面代码。

## 2. 实例项目最值得模仿的地方

实例项目的抽卡功能不是单个页面文件，而是一套小框架：

```text
主菜单入口
  |
  v
AppGacha                  // 抽卡流程页：待机、动画、结果
  |
  +--> sys_res             // 卡池资源读取，按星级建索引
  +--> sys_config          // 抽卡统计保存
  +--> AppGachaStats       // 统计页
  +--> lang/ui_strings     // 文案
  +--> hal                 // 显示、输入、音效、震动等硬件能力
```

其中最值得保留的思想有 4 个：

1. 抽卡结果先生成，动画只是展示。
2. 抽卡页面只管状态机、输入和绘制，不直接负责资源文件细节。
3. 统计数据不要只放页面局部变量，后续应交给 `sys_config` 管。
4. 卡池数据后续应从 `data` 读取，并由 `sys_res` 建好星级索引。

当前项目已经有了 App 框架，所以不要再把抽卡状态机写回 `main.cpp`。

## 3. 本项目推荐的目标结构

结合 `抽卡.md` 的“子界面显示：标准寻访 / 抽卡历史 / 返回”，建议这样拆：

```text
    抽卡规则、概率、保底、统计更新、卡池选择

src/sys/includes/sys/sys_res.h
```

第一版建议先实现前三个 App 文件和少量 `lang/ui` 常量。`sys_res`、`sys_config`、`data` 可以等流程稳定后再接。

## 4. 当前 app_gacha.cpp 的定位

当前项目里的 `src/apps/sources/app_gacha.cpp` 已经是一个 `AppMenuBase` 菜单页。

建议继续把它当“干员寻访入口菜单”，不要直接改成动画页。

推荐菜单：

```text
干员寻访
├── 标准寻访
├── 抽卡历史
└── 返回
```

对应行为：

```text
标准寻访 -> appManager.push(AppId::GachaStandard)
抽卡历史 -> appManager.push(AppId::GachaHistory)
返回     -> appManager.pop()
```

这样做的好处是：

- 入口菜单很干净。
- 抽卡流程页可以专心处理动画和结果。
- 历史/统计页可以独立迭代。
- 以后加“限定寻访”“规则说明”“卡池设置”时，只需要增加菜单项和 App。

## 5. 需要新增的 AppId

当前 `AppId` 只有：

```cpp
enum class AppId : uint8_t {
    Standby,
    MainMenu,
    Gacha,
};
```

建议扩展为：

```cpp
enum class AppId : uint8_t {
    Standby,
    MainMenu,
    Gacha,
    GachaStandard,
    GachaHistory,
};
```

同时在 `src/sys/sources/app_registry.cpp` 增加：

```cpp
extern AppBase *appGachaStandard;
extern AppBase *appGachaHistory;

case AppId::GachaStandard:
    return appGachaStandard;
case AppId::GachaHistory:
    return appGachaHistory;
```

规则和实例项目一致：页面跳转只认 `AppId`，不要让 App 之间互相保存裸指针。

## 6. 抽卡流程页的状态机

`app_gacha_standard.cpp` 建议继承 `AppBase`，因为它不是菜单，而是独立交互流程。

状态机：

```text
Idle      确认页，显示“标准寻访 / 短按开始”
Animating 动画页，显示 10 个结果格子逐个揭示
Result    结果页，列表显示十连结果
```

推荐枚举：

```cpp
enum class GachaPhase : uint8_t {
    Idle,
    Animating,
    Result,
};
```

页面输入：

| 阶段 | 短按 | 长按 | 旋钮 |
|---|---|---|---|
| Idle | 开始十连，生成结果并进入动画 | 暂不处理，或按框架习惯返回入口菜单 | 无 |
| Animating | 跳过动画，直接进入结果页 | 暂不处理 | 无 |
| Result | 返回干员寻访入口菜单 | 暂不处理 | 滚动十连结果 |

说明：

- `抽卡.md` 写的是“短按：返回，长按：无”，所以结果页优先按短按返回。
- 为了符合当前框架的统一习惯，也可以保留长按返回作为调试入口；如果严格按文档，长按就不做任何业务。

## 7. 抽卡结果必须先生成

流程应该是：

```text
短按开始
  |
  v
executeTenPull()
  |
  +--> 连续生成 10 个结果
  +--> 更新保底状态
  +--> 更新内存统计
  +--> 清空 revealed[10]
  |
  v
phase = Animating
  |
  v
动画逐个揭示已经确定的结果
  |
  v
phase = Result
```

不要在动画过程中实时抽卡。

这样可以保证：

- 跳过动画不会改变结果。
- 动画卡顿不会影响概率。
- 后续做统计保存时，保存点明确。

## 8. 抽卡数据结构建议

第一版可以先把卡池写死在代码里，后续再迁移到 `data`。

推荐结构：

```cpp
enum class GachaRarity : uint8_t {
    Star3 = 3,
    Star4 = 4,
    Star5 = 5,
    Star6 = 6,
};

struct GachaOperator {
    const char *name;
    GachaRarity rarity;
};

struct GachaPullResult {
    const GachaOperator *op;
    GachaRarity rarity;
    bool from_pity;
    bool standard_select_hit;
};

struct GachaStats {
    uint32_t total;
    uint32_t star6;
    uint32_t star5;
    uint32_t star4;
    uint32_t star3;
};

struct GachaPityState {
    uint16_t pulls_since_star6;
    uint16_t banner_pulls;
    bool first_standard_select_used;
    bool second_standard_select_used;

```

第一版最少需要：

- `GachaOperator`
- `GachaPullResult`
- `GachaStats`
- `pulls_since_star6`

`standard_select_hit` 和 150/300 抽标准选调可以先预留字段，后面再实现。

## 9. 概率规则建议

`抽卡.md` 中有一处概率需要注意：

- 前面星级列表写了“六星出率 8%、五星 50%、四星 40%”。
- 后面的标准寻访保底规则写了“六星概率从原本 2% 提升至 4%”。

为了贴近“标准寻访”的常见规则，建议第一版默认按下面实现：

```text
六星：2%
五星：8%
四星：50%
三星：40%
```

如果你确认要按 `抽卡.md` 前半部分的文字实现，只需要改概率常量，不要改页面代码。

推荐集中放常量：

```cpp
namespace GachaRates {
constexpr int kRateScale = 10000;
constexpr int kStar6Base = 200;    // 2%
constexpr int kStar5 = 800;        // 8%
constexpr int kStar4 = 5000;       // 50%
constexpr int kStar3 = 4000;       // 40%
constexpr int kPityStart = 50;
constexpr int kPityStep = 200;     // 每次 +2%
}
```

六星动态概率：

```text
如果 pulls_since_star6 < 50:
    six_rate = 2%
否则:
    six_rate = 2% + (pulls_since_star6 - 49) * 2%
    最高 100%
```

抽到六星后：

```text
pulls_since_star6 = 0
```

没有抽到六星：

```text
pulls_since_star6 += 1
```

## 10. 单抽流程建议

推荐把抽卡逻辑写成独立函数：

```cpp
GachaPullResult rollSingle()
{
    int six_rate = calcSixRate();
    int r = random(GachaRates::kRateScale);

    GachaRarity rarity;
    if (r < six_rate) {
        rarity = GachaRarity::Star6;
    } else if (r < six_rate + GachaRates::kStar5) {
        rarity = GachaRarity::Star5;
    } else if (r < six_rate + GachaRates::kStar5 + GachaRates::kStar4) {
        rarity = GachaRarity::Star4;
    } else {
        rarity = GachaRarity::Star3;
    }

    const GachaOperator *op = pickOperatorByRarity(rarity);
    updatePity(rarity);
    updateStats(rarity);

    return { op, rarity, false, false };
}
```

注意：

- `calcSixRate()`、`pickOperatorByRarity()`、`updatePity()`、`updateStats()` 后续都可以移动到 `sys_gacha.cpp`。
- 页面只调用 `executeTenPull()`，不要在绘制函数里计算概率。

## 11. 十连流程建议

`抽卡.md` 没有要求十连保底五星，只有标准寻访六星保底和 150/300 标准选调。

第一版十连可以只是连续 10 次 `rollSingle()`：

```cpp
static constexpr int kPullCount = 10;
GachaPullResult current_pulls[kPullCount];

void executeTenPull()
{
    for (int i = 0; i < kPullCount; ++i) {
        current_pulls[i] = rollSingle();
        revealed[i] = false;
    }
    scroll_offset = 0;
}
```

如果后续要加“十连至少五星”，也应该作为单独规则加在 `sys_gacha`，不要散落在页面里。

## 12. 标准选调 150/300 抽规则

`抽卡.md` 中的规则是：

- 本期标准寻访累计 150 次后，下次招募到的六星必定为本期出率上升的六星之一。
- 累计 300 次后，下次招募到的六星必定为本期出率上升的另一名六星。
- 这两个效果各限一次。
- 本期结束时清零。

这个规则依赖“本期 UP 六星是谁”，当前 `抽卡.md` 没有单独指定。

建议先预留：

```cpp
struct GachaBanner {
    const char *title;
    const GachaOperator *up6_a;
    const GachaOperator *up6_b;
};
```

第一版可以先不启用标准选调，只记录 `banner_pulls`。

后续启用时：

```text
每抽一次 banner_pulls += 1

如果本抽为六星：
    如果 banner_pulls >= 300 且第二次标准选调未使用:
        强制给 up6_b
        second_standard_select_used = true
    否则如果 banner_pulls >= 150 且第一次标准选调未使用:
        强制给 up6_a
        first_standard_select_used = true
    否则:
        正常从六星池随机
```

这个逻辑应放在 `rollSingle()` 或 `pickSixStarOperator()` 内，不要放在 UI 绘制里。


## 19. 标准寻访页骨架示意

下面只是结构示意，实际写代码时按当前 HAL 接口调整：

```cpp
class AppGachaStandard : public AppBase {
public:
    void onCreate() override
    {
        phase_ = GachaPhase::Idle;
        scroll_offset_ = 0;
        dirty_ = true;
    }

    void onLoop() override
    {
        if (phase_ == GachaPhase::Animating) {
            updateAnimation();
        }

        if (!dirty_) return;
        draw();
        dirty_ = false;
    }

    void onDestroy() override {}

    void onKnob(int delta) override
    {
        if (phase_ != GachaPhase::Result) return;
        scroll_offset_ = clampResultOffset(scroll_offset_ + delta);
        dirty_ = true;
    }

    void onKeyShort() override
    {
        if (phase_ == GachaPhase::Idle) {
            executeTenPull();
            phase_ = GachaPhase::Animating;
            anim_start_ms_ = millis();
        } else if (phase_ == GachaPhase::Animating) {
            phase_ = GachaPhase::Result;
        } else if (phase_ == GachaPhase::Result) {
            appManager.pop();
        }
        dirty_ = true;
    }

    void onKeyLong() override
    {
        // 抽卡.md 写长按无。调试期如需统一返回，可改为 appManager.pop();
    }

private:
    GachaPhase phase_ = GachaPhase::Idle;
    GachaPullResult pulls_[10];
    bool revealed_[10] = {};
    int scroll_offset_ = 0;
    uint32_t anim_start_ms_ = 0;
    bool dirty_ = true;
};
```

## 20. `app_gacha.cpp` 入口菜单示意

当前文件已经是菜单页，只需要把叶子点击接上：

```cpp
bool onLeafSelected(MenuItem *item) override
{
    if (item == &gacha_standard) {
        appManager.push(AppId::GachaStandard);
        return true;
    }

    if (item == &gacha_history) {
        appManager.push(AppId::GachaHistory);
        return true;
    }

    if (item == &gacha_back) {
        appManager.pop();
        return true;
    }

    return false;
}
```

如果某个菜单项只是说明页，可以不处理，让 `AppMenuBase` 进入默认内容页。

## 23. 不建议第一版做的事

第一版不要做：

- 不要接音效。
- 不要接震动。
- 不要做持久化。
- 不要做双语言。
- 不要一次录入全部干员。
- 不要一开始就接 LittleFS 和 ArduinoJson。
- 不要把完整卡池塞进 `main.cpp`。
- 不要让 `hal` 知道抽卡规则。
- 不要在绘制函数里更新概率和保底。

先让流程跑通，再逐步替换底层数据来源。

## 24. 最小验收标准

第一版做到下面这些就算完成：

- 能从主菜单进入“干员寻访”。
- “干员寻访”里有“标准寻访 / 抽卡历史 / 返回”。
- 进入“标准寻访”后，短按能开始十连。
- 十连结果在动画前已经固定。
- 动画页能逐个揭示 10 个格子。
- 动画中短按能跳过到结果页。
- 结果页用列表显示，每屏 5 条。
- 结果页旋钮能滚动查看 10 条结果。
- 结果页短按能返回入口菜单。
- 抽到六星后保底计数清零。
- 未抽到六星时保底计数增加。
- 统计页能看到总抽数和各星级数量。
- 重复进入、返回、再进入，不死机。
- `main.cpp` 不新增抽卡业务逻辑。

## 25. 后续扩展路线

完成第一版后，按这个顺序升级：

1. 把抽卡逻辑从 `app_gacha_standard.cpp` 抽到 `sys_gacha`。
2. 把完整干员名单整理成 `data/gacha_standard_zh.json`。
3. 新增 `sys_res`，启动时读取卡池并按星级建索引。
4. 新增 `sys_config`，保存统计和保底状态。
5. 实现 150/300 标准选调。
6. 增加清空统计功能。
7. 增加限定寻访或其他卡池。
8. 最后再加音效、震动、更多动画。

## 26. 总结

这次功能建议按“入口菜单 + 流程页 + 历史统计页 + 抽卡规则层 + 资源/配置层”的形状来做。

当前最合适的落点是：

```text
app_gacha.cpp             只做干员寻访入口菜单
app_gacha_standard.cpp    做标准寻访状态机
app_gacha_history.cpp     做历史和统计显示
sys_gacha.*               后续承接概率、保底、统计
sys_res.*                 后续承接卡池资源
sys_config.*              后续承接保存
```

这样既像实例项目，又不会一下子把当前项目抬得太复杂。后续你要换卡池、改概率、加持久化，都有明确位置可以改。
