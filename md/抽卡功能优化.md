# 抽卡功能优化

## 1. 优化目标

当前第一版“标准寻访”已经具备：

- 干员寻访入口菜单
- 标准寻访确认页、动画页、结果页
- 抽卡历史/统计页
- 运行期统计
- LittleFS + JSON 保存统计、保底和最近十连
- 六星 50 抽后每抽 +2% 的保底逻辑

下一步需要补齐标准寻访里的“两个 UP 六星角色”。

优化目标是：

- 标准寻访卡池中存在两个当前 UP 六星角色。
- 两个 UP 角色不要写死在页面代码里。
- UP 角色可以从配置中读取。
- UP 角色可以通过后续设置页修改。
- 修改后写入 `/config.json`，重启后仍然生效。
- 抽卡逻辑只放在 `sys_gacha`，页面只负责显示和修改配置。

## 3. 数据结构设计

建议在 `src/sys/includes/sys/sys_gacha.h` 中新增：

```cpp
struct Banner {
    const char *title;
    uint8_t up6_a_index;
    uint8_t up6_b_index;
};
```

含义：

```text
title       当前卡池标题，例如“标准寻访”
up6_a_index 第一个 UP 六星在六星池中的索引
up6_b_index 第二个 UP 六星在六星池中的索引
```

例如六星池：

```cpp
const Operator kStar6Pool[] = {
    { "鸿雪", Rarity::Star6 },     // index 0
    { "玛恩纳", Rarity::Star6 },   // index 1
    { "白铁", Rarity::Star6 },     // index 2
    { "斥罪", Rarity::Star6 },     // index 3
};
```

默认 UP：

```cpp
Banner g_banner = {
    "标准寻访",
    0, // 鸿雪
    1, // 玛恩纳
};
```

## 4. JSON 保存格式

当前项目已经使用 `/config.json` 保存抽卡数据。

建议扩展为：

```json
{
  "gacha": {
    "banner": {
      "title": "标准寻访",
      "up6_a_index": 0,
      "up6_b_index": 1
    },
    "stats": {
      "total": 0,
      "star6": 0,
      "star5": 0,
      "star4": 0,
      "star3": 0
    },
    "pity": {
      "pulls_since_star6": 0,
      "banner_pulls": 0,
      "first_standard_select_used": false,
      "second_standard_select_used": false
    },
    "last_pulls": []
  }
}
```

说明：

- `banner` 保存当前卡池设置。
- `up6_a_index` 和 `up6_b_index` 保存的是六星池索引，不保存指针。
- 不建议保存干员名字作为主要索引，因为以后名字变化或本地化时不好迁移。
- 可以额外保存 title，但真正决定 UP 的是 index。

```cpp
    const char *title = banner["title"] | "标准寻访";
```

```cpp

banner["up6_a_index"] = gacha_banner.up6_a_index;
banner["up6_b_index"] = gacha_banner.up6_b_index;
```

边界保护建议：

```cpp
gacha_banner.up6_a_index = SysGacha::ClampStar6Index(gacha_banner.up6_a_index);
gacha_banner.up6_b_index = SysGacha::ClampStar6Index(gacha_banner.up6_b_index);
```

推荐自动修正：

```cpp
if (gacha_banner.up6_a_index == gacha_banner.up6_b_index) {
    gacha_banner.up6_b_index = SysGacha::NextStar6Index(gacha_banner.up6_a_index);

```

## 6. sys_gacha 接口设计

在 `src/sys/includes/sys/sys_gacha.h` 中新增接口：

```cpp
const Banner &GetBanner();
void SetBannerUp6(uint8_t up_a_index, uint8_t up_b_index);

int GetStar6PoolCount();
const Operator *GetStar6Operator(int index);
uint8_t ClampStar6Index(uint8_t index);
uint8_t NextStar6Index(uint8_t index);
```

用途：

- 设置页用 `GetStar6PoolCount()` 知道可选范围。
- 设置页用 `GetStar6Operator(index)` 显示干员名。
- 设置页用 `SetBannerUp6()` 修改当前 UP。
- `SetBannerUp6()` 内部负责同步到 `sysConfig` 并保存。

示例实现：

```cpp
int GetStar6PoolCount()
{
    return sizeof(kStar6Pool) / sizeof(kStar6Pool[0]);
}

const Operator *GetStar6Operator(int index)
{
    if (index < 0) index = 0;
    if (index >= GetStar6PoolCount()) index = GetStar6PoolCount() - 1;
    return &kStar6Pool[index];
}

uint8_t ClampStar6Index(uint8_t index)
{
    int count = GetStar6PoolCount();
    if (count <= 0) return 0;
    if (index >= count) return count - 1;
    return index;
}

uint8_t NextStar6Index(uint8_t index)
{
    int count = GetStar6PoolCount();
    if (count <= 1) return 0;
    return (index + 1) % count;
}
```

`LoadStateFromConfig()` 中同步 banner：

```cpp
g_banner.up6_a_index = ClampStar6Index(sysConfig.gacha_banner.up6_a_index);
g_banner.up6_b_index = ClampStar6Index(sysConfig.gacha_banner.up6_b_index);

if (g_banner.up6_a_index == g_banner.up6_b_index) {
    g_banner.up6_b_index = NextStar6Index(g_banner.up6_a_index);
}
```

`SetBannerUp6()`：

```cpp
void SetBannerUp6(uint8_t up_a_index, uint8_t up_b_index)
{
    g_banner.up6_a_index = ClampStar6Index(up_a_index);
    g_banner.up6_b_index = ClampStar6Index(up_b_index);

    if (g_banner.up6_a_index == g_banner.up6_b_index) {
        g_banner.up6_b_index = NextStar6Index(g_banner.up6_a_index);
    }

    sysConfig.gacha_banner.up6_a_index = g_banner.up6_a_index;
    sysConfig.gacha_banner.up6_b_index = g_banner.up6_b_index;
    sysConfig.save();
}
```

## 7. 抽卡逻辑如何使用 UP

当前第一版可以先只实现 150/300 标准选调，不做复杂 UP 概率。

逻辑：

```text
每抽一次：
  banner_pulls += 1

如果本抽不是六星：
  正常返回

如果本抽是六星：
  如果 banner_pulls >= 300 且第二次标准选调未使用：
      强制给 UP 2
      标记 second_standard_select_used = true
  否则如果 banner_pulls >= 150 且第一次标准选调未使用：
      强制给 UP 1
      标记 first_standard_select_used = true
  否则：
      正常从六星池随机
```

推荐把六星干员选择拆成函数：

```cpp
const Operator *pickSixStarOperator()
{
    if (g_pity.banner_pulls >= 300 && !g_pity.second_standard_select_used) {
        g_pity.second_standard_select_used = true;
        return GetStar6Operator(g_banner.up6_b_index);
    }

    if (g_pity.banner_pulls >= 150 && !g_pity.first_standard_select_used) {
        g_pity.first_standard_select_used = true;
        return GetStar6Operator(g_banner.up6_a_index);
    }

    return pickFromPool(kStar6Pool);
}
```

注意：

`banner_pulls` 应该在判断标准选调前已经加 1，表示当前这一抽已经计入本期累计。

`rollSingle()` 中：

```cpp
const Operator *op = nullptr;
if (rarity == Rarity::Star6) {
    op = pickSixStarOperator();
} else {
    op = pickOperatorByRarity(rarity);
}
```

然后再更新统计和六星保底。

## 8. 是否需要 UP 概率

标准寻访通常还会有“UP 六星出率提升”的概念。

第一版可以先不做，只做 150/300 标准选调，因为当前文档明确提到了 150/300，但没有明确普通六星命中 UP 的比例。

如果后续要做，推荐加常量：

```cpp
namespace Rates {
constexpr int kUp6Rate = 5000; // 50%
}
```

六星时：

```text
先判断 150/300 标准选调
如果没有触发：
  50% 命中 UP 之一
  50% 从全部六星或非 UP 六星里随机
```

简单实现：

```cpp
if (random(Rates::kRateScale) < Rates::kUp6Rate) {
    return random(2) == 0
        ? GetStar6Operator(g_banner.up6_a_index)
        : GetStar6Operator(g_banner.up6_b_index);
}

return pickFromPool(kStar6Pool);
```

如果不希望普通随机再次抽到 UP，可以后续实现 `pickNonUpSixStarOperator()`。

## 17. 最小验收标准

完成后应满足：

- 标准寻访确认页能看到当前两个 UP 六星。
- 可以从“干员寻访”进入“卡池设置”。
- 可以修改 UP1 和 UP2。
- 修改后回到标准寻访确认页能看到新 UP。
- 重启后 UP 设置仍然存在。
- 十连完成后统计仍然能保存。
- 150 抽后下一个六星会强制给 UP1。
- 300 抽后下一个六星会强制给 UP2。
- 抽卡业务逻辑没有写进 `main.cpp`。

## 18. 注意点

1. 不要保存 `const Operator *` 指针。

指针只在本次运行中有效，写进 JSON 没意义。

2. 保存索引比保存名字更稳定。

例如：

```json
"up6_a_index": 0
```

比：

```json
"up6_a": "鸿雪"
```

更适合当前第一版。

3. 修改卡池顺序会影响旧配置。

如果以后完整卡池从 JSON 加载，最好给每个干员加稳定 ID。

例如：

```json
{ "id": "arknights_horny_hongxue", "name": "鸿雪", "rarity": 6 }
```

到那时 UP 配置应从 index 升级为 id。

4. 当前第一版先用 index 足够。

因为六星池还写死在代码里，顺序可控，实现简单。

## 19. 总结

这次优化的核心是：

```text
UP 角色属于卡池配置，不属于页面状态。
```

正确分层是：

```text
sys_gacha
    管卡池、UP、概率、保底、标准选调

sys_config
    管 UP 设置和抽卡状态保存

app_gacha_banner_settings
    管用户如何修改 UP

app_gacha_standard
    只显示当前 UP，并调用抽卡流程
```

这样后续要换 UP、换卡池、加限定寻访，都不用推翻标准寻访页面。

## 20. 后续新增角色应该如何做

新增角色分两个阶段看。

当前第一版卡池还是写死在代码里，所以新增角色要改 `src/sys/sources/sys_gacha.cpp` 里的静态数组。

未来如果卡池迁移到 JSON，就应该改 `data/gacha_standard_zh.json`，再由 `sys_res` 加载。

### 20.1 当前第一版：在代码数组里新增

当前卡池类似这样：

```cpp
const Operator kStar6Pool[] = {
    { "鸿雪", Rarity::Star6 },
    { "玛恩纳", Rarity::Star6 },
    { "白铁", Rarity::Star6 },
    { "斥罪", Rarity::Star6 },
};
```

如果要新增一个六星角色，例如“新六星”，就在 `kStar6Pool` 后面追加：

```cpp
const Operator kStar6Pool[] = {
    { "鸿雪", Rarity::Star6 },
    { "玛恩纳", Rarity::Star6 },
    { "白铁", Rarity::Star6 },
    { "斥罪", Rarity::Star6 },
    { "新六星", Rarity::Star6 },
};
```

五星、四星、三星同理，分别添加到：

```cpp
kStar5Pool
kStar4Pool
kStar3Pool
```

新增后需要：

```text
1. 编译固件
2. 烧录设备
3. 进入卡池设置页
4. 如果需要，把 UP1/UP2 改成新角色
```

### 20.2 重要注意：不要随便调整已有角色顺序

当前配置里保存 UP 角色用的是六星池索引：

```json
"up6_a_index": 0,
"up6_b_index": 1
```

这意味着：

```text
0 = 鸿雪
1 = 玛恩纳
2 = 白铁
3 = 斥罪
```

如果你把数组顺序改了，旧配置会指向不同角色。

例如原来：

```cpp
0 鸿雪
1 玛恩纳
```

旧配置：

```json
"up6_a_index": 0
```

表示 UP1 是鸿雪。

如果你改成：

```cpp
0 新六星
1 鸿雪
2 玛恩纳
```

旧配置里的 `0` 就会变成“新六星”。

所以当前第一版新增角色建议：

```text
只在数组末尾追加新角色
不要插入到中间
不要重排已有角色
不要删除已有角色
```

如果确实要删除角色，建议同时清理或重置 `/config.json` 里的 UP 索引。

### 20.3 当前第一版新增角色后的代码影响

如果只是追加角色，通常不需要改其他逻辑。

这些函数会自动适配数组长度：

```cpp
GetStar6PoolCount()
GetStar6Operator(index)
ClampStar6Index(index)
NextStar6Index(index)
pickFromPool(kStar6Pool)
```

原因是它们应该通过数组长度计算：

```cpp
sizeof(kStar6Pool) / sizeof(kStar6Pool[0])
```

所以只要角色加进数组，抽卡和卡池设置页就能看到。

### 20.4 保存最近十连时的影响

当前建议保存最近十连时保存：

```json
{
  "rarity": 6,
  "pool_index": 0
}
```

这也依赖数组顺序。

如果只是末尾追加角色，旧历史不会受影响。

如果重排数组，旧历史显示的角色也可能变化。

所以当前第一版仍然建议：

```text
新增角色只追加，不重排。
```

### 20.5 未来正式方案：给每个角色稳定 ID

等后续接 JSON 卡池时，不建议继续只靠 index。

更推荐每个角色有稳定 ID：

```json
[
  {
    "id": "arknights_hongxue",
    "name": "鸿雪",
    "rarity": 6
  },
  {
    "id": "arknights_mlynar",
    "name": "玛恩纳",
    "rarity": 6
  }
]
```

UP 配置保存 ID：

```json
{
  "gacha": {
    "banner": {
      "up6_a_id": "arknights_hongxue",
      "up6_b_id": "arknights_mlynar"
    }
  }
}
```

最近十连也保存 ID：

```json
{
  "rarity": 6,
  "operator_id": "arknights_hongxue"
}
```

这样就算以后卡池文件重排：

```text
鸿雪从第 0 个移动到第 20 个
```

配置依然能正确找到鸿雪。

### 20.6 未来 JSON 卡池建议格式

建议文件：

```text
data/gacha_standard_zh.json
```

格式：

```json
[
  { "id": "arknights_hongxue", "name": "鸿雪", "rarity": 6 },
  { "id": "arknights_mlynar", "name": "玛恩纳", "rarity": 6 },
  { "id": "arknights_baitie", "name": "白铁", "rarity": 6 },
  { "id": "arknights_xiaoge", "name": "晓歌", "rarity": 5 },
  { "id": "arknights_yeyan", "name": "夜烟", "rarity": 4 },
  { "id": "arknights_fang", "name": "芬", "rarity": 3 }
]
```

`sys_res` 负责：

```text
读取 JSON
解析所有角色
按 rarity 建索引
提供按星级随机抽取
提供按 id 查找角色
```

`sys_gacha` 负责：

```text
计算概率
处理保底
处理 UP
向 sys_res 请求角色
```

### 20.7 未来从 index 升级到 id 的迁移策略

如果设备上已经有旧配置：

```json
"up6_a_index": 0
```

后续升级到：

```json
"up6_a_id": "arknights_hongxue"
```

可以在 `sys_config.load()` 里兼容旧字段：

```cpp
if (banner["up6_a_id"].is<const char *>()) {
    // 新配置：按 id 读取
} else {
    // 旧配置：按 index 读取，再转成当前角色 id
}
```

迁移后 `save()` 只写新字段。

这样用户升级固件后，原来的 UP 设置不会丢。

### 20.8 新增角色的推荐流程总结

当前第一版：

```text
1. 打开 src/sys/sources/sys_gacha.cpp
2. 找到对应星级数组
3. 在数组末尾追加新角色
4. 编译并烧录
5. 如需设为 UP，在卡池设置页选择它
```

未来 JSON 版：

```text
1. 打开 data/gacha_standard_zh.json
2. 新增一条角色记录，包含 id/name/rarity
3. 上传 LittleFS 数据
4. 重启设备或重新加载资源
5. 在卡池设置页选择它
```
