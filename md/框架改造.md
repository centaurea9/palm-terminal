# 框架改造

## 1. 这份文档的目标

这份文档回答两个问题：

1. 你当前项目和实例工程的框架差别到底在哪里。
2. 如果你想把当前项目逐步改成实例工程那种结构，应该按什么顺序做。

重点不是“一次性照抄实例工程全部代码”，而是先做出一个适合你当前仓库体量的最小框架，再逐步向实例工程靠拢。

## 2. 当前项目和实例工程的本质区别

### 2.1 你当前项目

你现在的工程更接近“单体式菜单程序”：

- `main.cpp` 里同时包含菜单状态、输入处理、动画推进、绘制和页面切换逻辑
- `menu.h / menu.cpp` 保存菜单树
- `screen/` 负责显示
- `input/` 负责 EC11

优点：

- 简单直接
- 改动快
- 所有逻辑都能在一个地方看到

缺点：

- 功能一多，`main.cpp` 会越来越重
- 页面之间没有统一生命周期
- 菜单、功能页、系统页很难扩展成统一架构
- 抽卡、统计、设置、待机这些功能后面会越来越耦合

### 2.2 实例工程

实例工程更接近“分层式 App 框架”：

- `main.cpp` 只负责系统初始化和主循环
- 页面统一走 `AppBase`
- 页面跳转统一走 `AppManager`
- 页面注册统一走 `AppRegistry`
- 输入和显示统一通过 `HAL`
- 配置、资源、文件系统统一收进 `sys`
- 通用布局、主题、字体统一收进 `ui`
- 文案和语言切换统一收进 `lang`

优点：

- 结构清晰
- 后期增加页面更稳
- 各模块职责边界明确
- 更适合抽卡、统计、设置、待机这类持续扩展

缺点：

- 文件会变多
- 改造前期成本更高
- 一开始会感觉“绕了一层”

## 3. 你真正该学实例工程的是什么

不要直接照抄它所有模块。

你最该学的是下面 6 个框架思想：

1. `main.cpp` 变薄  
只保留初始化和主循环。

2. 页面统一生命周期  
每个页面都实现统一接口。

3. 页面跳转统一管理  
不要页面互相直接调用。

4. 硬件和业务分离  
显示、输入、按键都统一走 `HAL`。

5. 共享 UI 抽出来  
菜单布局、字体度量、主题参数不要散在页面里。

6. 配置和资源独立  
卡池、统计、配置以后都不要绑死在页面代码里。

## 4. 推荐的目标目录结构

结合你当前项目规模，建议目标结构先做成“缩小版实例工程”：

```text
src/
├─ main.cpp
├─ apps/
│  ├─ includes/apps/
│  │  └─ app_menu_base.h
│  └─ sources/
│     ├─ app_main_menu.cpp
│     ├─ app_standby.cpp
│     └─ app_gacha.cpp
├─ hal/
│  ├─ includes/hal/
│  │  └─ hal.h
│  └─ sources/
│     └─ hal.cpp
├─ sys/
│  ├─ includes/sys/
│  │  ├─ app_base.h
│  │  ├─ app_manager.h
│  │  ├─ app_registry.h
│  │  ├─ sys_constants.h
│  │  ├─ sys_fs.h
│  │  ├─ sys_config.h
│  │  └─ sys_res.h
│  └─ sources/
│     ├─ app_manager.cpp
│     ├─ app_registry.cpp
│     ├─ sys_fs.cpp
│     ├─ sys_config.cpp
│     └─ sys_res.cpp
├─ lang/
│  └─ includes/lang/
│     ├─ terminal_lang.h
│     └─ ui_strings.h
├─ ui/
│  └─ includes/ui/
│     ├─ ui_theme.h
│     └─ ui_font_config.h
├─ input/                 // 过渡阶段可先保留
├─ screen/                // 过渡阶段可先保留
└─ u8g2_wrapper/
```

## 5. 当前文件到目标结构的迁移关系

| 当前文件 | 目标位置 | 说明 |
|---|---|---|
| `src/main.cpp` | 仍保留 `src/main.cpp` | 但只保留系统入口职责 |
| `src/menu.h` | 拆进 `apps` / `lang` | 不再作为全局静态菜单树中心 |
| `src/menu.cpp` | `apps/sources/app_main_menu.cpp` | 主菜单变成页面对象 |
| `src/input/ec11.*` | `hal/` | 编码器能力统一收进 HAL |
| `src/screen/screen.*` | `hal/` + `ui/` | 显示能力和字体度量逐步迁出 |
| `src/u8g2_wrapper/*` | 暂时保留 | 作为底层依赖继续使用 |

## 6. 改造时哪些先做，哪些后做

### 6.1 第一批必须做

- `sys/app_base.*`
- `sys/app_manager.*`
- `sys/app_registry.*`
- `sys/sys_constants.*`
- `hal/hal.*`
- `apps/app_main_menu.cpp`
- `apps/app_standby.cpp`

### 6.2 第二批建议做

- `ui/ui_theme.h`
- `apps/app_menu_base.h`
- `lang/ui_strings.h`

### 6.3 第三批再做

- `sys/sys_fs.*`
- `sys/sys_config.*`
- `sys/sys_res.*`
- `data/*`

### 6.4 当前先不要着急做

- `sys_network`
- `sys_ble`
- `sys_nfc`
- `sys_auto_push`
- `sys_event`
- `bsp`

这些不是“先把框架立起来”的第一优先级。

## 7. 你应该按多少步完成

推荐按 12 步完成。

每一步都尽量做到：

- 能编译
- 能运行
- 行为可验证

再进入下一步。

## 8. 第 1 步：冻结当前可运行版本

要做的事：

- 确认当前工程能完整编译
- 记录当前菜单和输入行为
- 最好留一个基准提交

这一步的目标：

- 后面拆坏时知道从哪里回退

验收标准：

- 菜单能滚动
- 短按能进入内容态
- 长按能返回

## 9. 第 2 步：先建目标目录，不改业务

要做的事：

- 新建 `src/apps`
- 新建 `src/hal`
- 新建 `src/sys`
- 新建 `src/lang`
- 新建 `src/ui`

这一步的目标：

- 给后面迁移代码腾位置

验收标准：

- 工程还能编译
- 当前行为不变

## 10. 第 3 步：建立 `sys_constants.h`

先把现在散在项目里的关键常量收进一个地方。

优先收这些：

- 屏幕宽高
- 引脚号
- 长按时间
- 菜单栈深度
- 刷新间隔
- 菜单布局常量

建议优先迁这些常量：

- `EC11_PIN_A/B/SW`
- `EC11_LONG_PRESS_MS`
- `MENU_STACK_MAX`
- `FRAME_FAST_MS`
- `MENU_DIVIDER_X`
- `MENU_BOX_H`
- `MENU_ITEM_SPACING_Y`

这一步的目标：

- 先清掉 `main.cpp` 里的魔法数字

验收标准：

- 行为不变
- 只是常量位置改变

## 11. 第 4 步：建立最小 `HAL`

第一版 HAL 不求完整，只求把调用入口统一。

建议先做这些接口：

- `HAL_Init()`
- `HAL_Get_Knob_Delta()`
- `HAL_Get_Click()`
- `HAL_Get_LongPress()`
- `HAL_Screen_Clear()`
- `HAL_Screen_Update()`
- `HAL_Draw_Text()`
- `HAL_Get_Text_Width()`
- `HAL_Get_Screen_Width()`
- `HAL_Get_Screen_Height()`

注意：

- 第一版 HAL 内部可以继续转调你原来的 `screen_*`
- 第一版 HAL 内部也可以继续使用 `EC11Input`

重点是先统一入口，不是马上重写底层。

验收标准：

- `main.cpp` 不再直接调用 `screen_*`
- `main.cpp` 不再直接操作 `EC11Input`

## 12. 第 5 步：先把菜单相关代码从 `main.cpp` 拆出来

此时还不一定要正式变成 `App`，但至少先把下面这些内容迁出：

- 菜单动画推进
- 菜单绘制
- 菜单输入响应
- 菜单内容态显示

这一步的目标：

- 先把 `main.cpp` 降重

验收标准：

- 当前菜单行为和拆分前一致
- `main.cpp` 明显变短

## 13. 第 6 步：建立最小 `AppBase`

从这里开始，页面不再只是状态判断，而是独立对象。

第一版最小接口建议：

- `onCreate()`
- `onResume()`
- `onLoop()`
- `onDestroy()`
- `onKnob(int delta)`
- `onKeyShort()`
- `onKeyLong()`

这一步的目标：

- 从“状态机堆在一起”开始转向“页面对象”

验收标准：

- 至少能定义一个页面类并编译通过

## 14. 第 7 步：把主菜单改造成 `AppMainMenu`

这里是第一次真正进入实例工程风格。

要做的事：

- 新建 `apps/sources/app_main_menu.cpp`
- 把原来 `menu.cpp + main.cpp` 里的主菜单行为搬进去

重要提醒：

- 这一阶段可以保留你现在的菜单 UI
- 不需要立刻重写成实例工程完整 `AppMenuBase`

这一步的目标：

- 让主菜单正式成为一个页面对象

验收标准：

- 主菜单能独立响应旋钮、短按、长按
- 菜单逻辑不再写在 `main.cpp`

## 15. 第 8 步：建立最小 `AppRegistry`

要做的事：

- 定义 `AppId`
- 建立 `AppRegistry_Get(AppId)`
- 注册最小页面集合

第一批推荐只注册：

- `Standby`
- `MainMenu`

如果此时你已经开始做抽卡页，也可以加上：

- `Gacha`

这一步的目标：

- 页面开始有统一注册入口

验收标准：

- 能通过 `AppId` 拿到对应页面对象

## 16. 第 9 步：建立最小 `AppManager`

第一版 `AppManager` 不需要很复杂，只做核心调度就够了：

1. 保存当前页面指针
2. 调用当前页面 `onLoop()`
3. 从 HAL 取输入事件
4. 把输入事件分发给当前页面
5. 提供 `launch()`、`push()`、`pop()`

这一步的目标：

- 把页面调度彻底从 `main.cpp` 中拿出去

验收标准：

- `main.cpp` 不再直接判断当前页面状态
- 输入分发已经由 `AppManager` 接管

## 17. 第 10 步：把 `main.cpp` 瘦成系统入口

理想目标是：

`setup()` 只保留：

- 串口初始化
- `HAL_Init()`
- `appManager.begin()`

`loop()` 只保留：

- `appManager.run()`
- `delay(1)`

这一步的目标：

- 完成从“单体主循环”到“框架主循环”的第一次转型

验收标准：

- `main.cpp` 不再包含菜单绘制函数
- `main.cpp` 不再包含页面状态变量

## 18. 第 11 步：增加 `AppStandby`

待机页是很适合尽早独立出来的页面。

第一版可以很简单：

- 只显示一张简单待机画面
- 短按进入主菜单

先不要急着做这些复杂行为：

- 真实休眠
- 背光控制
- 唤醒恢复

这一步的目标：

- 先把“开机页”和“主菜单页”分开

验收标准：

- 开机先进入待机页
- 待机页能进入主菜单

## 19. 第 12 步：再抽 `AppMenuBase`、`ui_theme`、`lang`

当你至少有两个以上菜单类页面时，再做这一层最合适。

建议顺序：

1. `ui_theme.h`
2. `app_menu_base.h`
3. `ui_strings.h`

这一步的目标：

- 真正靠近实例工程的“共用框架”形态

验收标准：

- 至少两个页面共用同一套菜单绘制逻辑
- 文案不再散落在每个页面代码里

## 20. 抽卡功能应该插在哪一步

最合适的是放在第 10 步之后。

原因很简单：

- 在第 10 步前，主框架还没稳定
- 抽卡页会进一步放大页面切换、输入分发和绘制耦合

推荐顺序：

1. 先把 `MainMenu + Standby + AppManager` 跑稳
2. 再开始做 `AppGacha`
3. 最后再接统计、配置、资源池

## 21. 文件系统和配置应该插在哪一步

最合适的是放在第 12 步之后，或者和第 12 步并行推进一部分。

推荐顺序：

1. `sys_fs` 先只做 `LittleFS.begin()`
2. `sys_config` 先只存少量配置
3. `sys_res` 后面再做卡池和资源加载

不要一开始就把：

- LittleFS
- ArduinoJson
- 卡池 JSON
- 统计持久化

一起堆进来。

## 22. 每一步之间建议怎么停

建议不是一口气做完，而是每做完一组关键步骤就停一下验证。

推荐停点：

1. 做完第 3 步，停一次
2. 做完第 4 步，停一次
3. 做完第 7 步，停一次
4. 做完第 9 步，停一次
5. 做完第 10 步，停一次

每次都检查：

- 能不能编译
- 菜单还能不能用
- 旋钮方向有没有反
- 短按长按有没有错乱
- 返回逻辑有没有坏

## 23. 现阶段最不建议你的做法

这些会让改造过程很容易失控：

1. 一开始就全量照抄实例工程  
你的硬件、屏幕、当前工程体量都不一样。

2. 一开始就把 `bsp + hal + sys + ui + apps + lang + data` 全部能力建齐  
目录可以先建，但能力不要同时全上。

3. 一开始就把网络、BLE、NFC 一起迁入  
这会把调试面放大很多倍。

4. 一开始就推翻当前菜单 UI  
先搬框架，再换 UI，更稳。

## 24. 我最推荐你现在的切入点

如果你现在就准备开始，我最推荐先完成下面这 4 步：

1. 建目录骨架
2. 建 `sys_constants.h`
3. 建最小 `HAL`
4. 把主菜单搬成 `AppMainMenu`

这 4 步一完成，你的项目就已经从“当前结构”开始向“实例工程结构”转了。

后面的：

- `AppRegistry`
- `AppManager`
- `Standby`
- `Gacha`

就会顺很多。

## 25. 最后总结

把当前项目调整成实例工程那样，最合理的路线不是一次性重写，而是：

1. 先统一常量
2. 再统一硬件入口
3. 再把主菜单做成页面对象
4. 再引入页面注册和页面调度
5. 最后再抽共用 UI、配置和资源层

对你当前仓库来说，最重要的不是“先把抽卡搬进来”，而是先把下面这条主线跑通：

- `Standby`
- `MainMenu`
- `AppManager`

等这条主线稳定之后，再继续加：

- `Gacha`
- `GachaStats`
- `sys_fs`
- `sys_config`
- `sys_res`
